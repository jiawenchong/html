# 空調系統預算案件審核

## 為什麼這樣設計

審核規則一共 54 條，分散在四個類別（歷史案件 40 / 料單 3 / ROI 3 / 其他 8），
資料放在 `data/*.json`。**不要把全部 54 條塞進對話一次性背起來再憑印象審案** ——
對話一拉長、或被自動摘要，細節會跑掉，而且 54 條一次塞進 context 對單一案件來說
是噪音，容易出現「lost in the middle」漏看關鍵規則。

正確流程：每審一個新案件，都重新對 `scripts/retrieve_rules.py` 跑一次檢索，
讓規則從檔案重新讀進來，而不是依賴對話歷史裡還記得多少。Tier 1（下方核心鐵律）
篇幅小，直接寫在這裡，每次 skill 被讀取都會整段重新載入。

## Tier 1：核心鐵律（每案必查，不需檢索，直接套用）

逐項檢查，任何一項命中「退件條件」就標記退件並引用對應理由；全部通過才能放行：

1. **預審程序**：是否已完成 TQM 預審 / 依機制送審？未做 → 退件（程序性硬傷）。
2. **關鍵文件**：是否檢附完整料單（BOM）、設備規格（型號、RT、COP、冷媒類型、電壓等關鍵參數）、廠務專業評估報告？缺一即退件。
3. **圖文一致性**：改善作法的文字描述與圖示/設備類型是否一致（例如不可同時出現吊式FCU又註解分離式冷氣）？衝突即退件。
4. **備援/可靠性風險**：是否涉及高故障率設備或單點系統（SPOF）？若有，是否規劃備機/N+1/雙機備援？未評估風險即退件；有評估且降低風險是加分項。
5. **減碳策略**：設備是否使用高 GWP 冷媒（如 R-410A, GWP>2000）或低能效（氣冷式冰機 COP<3.0）？不符減碳策略即退件；引用 BKM Table 標準是最常見的通過理由。
6. **效益量化**：
   - 工安/良率類案件 → 可用 FMEA RPN 改善前後對比替代 ROI（不計 ROI 可接受，但必須有 RPN 數據）。
   - 其他所有案件 → 必須有具體節電量(kWh)、金額(NTD)或減碳量(tonCO2e)等量化數據；只寫「不計ROI」且無其他量化指標即退件。
7. **ROI 計算完整性**：分母（成本）是否涵蓋全生命週期——含設備拆除/安裝延伸工程費用、舊設備殘值（未滿攤提年限）？分母（效益）是否已扣除運轉電費及保養維護費用（即沒有把這些當作淨節省）？是否用夏季峰值節電量直接放大為全年效益（未考慮季節負載變動）？任一項有問題即退件。
8. **效益基準線**：節電/節碳效益是否有歷史用電基準線、實測數據或計算公式？只給結論數字、無基準線與計算式即退件。若現場已裝變頻器，改善前基線必須用「實測運轉功率」，不可用「馬達額定功率」估算（會高估效益）。
9. **預算合理性**：金額是否有佐證（前案 PO 單價 / BKM 標準 / 市場行情比價）？明顯高於市場行情且無依據即退件。
10. **料單比例**：非標準材料（如五金）金額是否超過材料金額 50%？工費比例是否超過總價 50%（純塑膠配管排除）？超過即退件。
11. **環境量測類案件**（溫度、噪音/振動、換氣量等改善方案）：是否有改善前後的量測數據與法規/規範門檻對比？只給結論（如「建議加裝隔音牆/冷風機」）卻無量測數據即退件。
12. **馬達/風扇降功率節電案**：是否檢附風機/馬達特性曲線（P-Q 曲線）、現場實測風量？是否確認仍滿足法定換氣量等硬性規範下限？是否已優先評估變頻（VFD）方案而非直接降規格？任一缺漏即退件傾向。

## Tier 2：執行步驟（每個案件都要做，不可省略）

1. 將案件的關鍵描述（標題、改善作法、設備規格、料單摘要、效益/ROI 計算說明）整理成一段文字。
2. 執行檢索腳本，取得匹配規則：
   ```
   python3 .claude/skills/hvac-budget-review/scripts/retrieve_rules.py "<案件關鍵描述>" --top 5
   ```
   也可以用 `--category history|material|roi|other|external` 只查單一類別，或 `--list-categories` 確認資料量。
3. 對照 Tier 1 鐵律逐項檢查（步驟見上）。
4. 整合 Tier 1 結果與檢索回來的 Tier 2 規則（含 id），形成審核結論。

## 輸出格式要求

審核結論必須：
- 明確給出「通過 / 退件 / 有條件通過」。
- **每一個判斷理由都要引用規則 id**（例如 `[HIST-09] ROI計算邏輯錯誤` 或 Tier 1 的鐵律編號），不可只憑印象敘述理由而不指出依據。
- 若退件，列出具體缺漏項目與建議補件方向（可參考該規則的「備註」欄位）。
- 若案件內容在 54 條規則中**完全沒有命中**任何關鍵字（腳本輸出「無匹配規則」），不要自行编造判斷依據——明確標註「未匹配標準規則，建議人工複核」，並列出案件中你判斷可能相關但規則庫未覆蓋的疑慮點。

## 資料與腳本

- `data/historical_cases.json`（40 筆：歷史退件/通過案例與理由）
- `data/material_list_rules.json`（3 筆：料單審查規則）
- `data/roi_rules.json`（3 筆：ROI 計算審查規則）
- `data/other_rules.json`（8 筆：特定設備/情境審查標準，如冰水泵、停車場風機、無塵室智能化、冷卻水塔、管路工程等）
- `data/external_data.json`（目前為空，預留給未來的外部資料）
- `scripts/retrieve_rules.py`（關鍵字檢索腳本，每筆規則用其 `keywords` 欄位做比對）

新增規則時，直接在對應的 `data/*.json` 加一筆記錄（含 `title`/`keywords`/`rule`/`example`/`disposition`/`note`），不需要改腳本或本文件。
